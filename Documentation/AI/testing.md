# BRAINSTools Testing

## Build type policy for testing

**Always run tests in the Release inner build.**  Debug builds are 10–20×
slower for numerically intensive ITK/ANTs algorithms and will cause nearly
all BCD, BRAINSABC, and registration tests to time out.

| Situation | Build to use |
|-----------|-------------|
| Routine test runs, CI verification | `BRAINSTools-Release-EPRelease-build` |
| Diagnosing a *specific* failing test | `BRAINSTools-Debug-EPRelease-build` |
| A test passes Release but fails Debug | Do not investigate — this is expected |

Use the Debug build only when a test fails in Release and you need a debugger
or AddressSanitizer to find the root cause.  Once the fix is verified in
Release, do not re-run the full suite in Debug.

## Running Tests

Tests must be run from the **inner product build directory**, not the top-level
SuperBuild directory. The inner build is named
`<build>/BRAINSTools-<type>-EPRelease-build/`.

### Preferred: BRAINSTools_run_tests.sh

Use the provided script, which scales parallel jobs to avoid oversubscription.
Tests declare `PROCESSORS 3` in CMakeLists so CTest manages scheduling natively;
the script defaults to `hw_cpus / 3` parallel jobs.

```bash
# Normal test run — always use the Release build
cd build/BRAINSTools-Release-EPRelease-build
../../Utilities/misc/BRAINSTools_run_tests.sh              # defaults: -j hw_cpus/3
../../Utilities/misc/BRAINSTools_run_tests.sh -j 4         # explicit parallel jobs
../../Utilities/misc/BRAINSTools_run_tests.sh -r BRAINSFit # run only BRAINSFit tests
../../Utilities/misc/BRAINSTools_run_tests.sh -- --rerun-failed

# Diagnosing a specific failing test — Debug build only
cd build/BRAINSTools-Debug-EPRelease-build
../../Utilities/misc/BRAINSTools_run_tests.sh -r FailingTestName
```

Options:
- `-j <N>` — parallel ctest jobs (default: `hw_cpus / 3`)
- `-r <R>` — ctest `-R` regex filter
- `--`     — pass remaining args directly to ctest

Each run saves a timestamped JUnit XML and text log under `TestResults/` in
the build directory (requires CMake ≥ 3.21 for `--output-junit`):

```
TestResults/
├── 20260401T103045.junit.xml   # machine-readable pass/fail per test
├── 20260401T103045.log         # full ctest text output
├── latest.junit.xml -> 20260401T103045.junit.xml
└── latest.log       -> 20260401T103045.log
```

To fix failing tests one at a time without re-running everything:
```bash
../../Utilities/misc/BRAINSTools_run_tests.sh -- --rerun-failed
```

### Generating a test report

`bt-report-tests.sh` parses any CTest log and prints a structured report:
summary counts, wall-clock time, top-N slowest tests, and a failure table.
It is called automatically at the end of every `BRAINSTools_run_tests.sh` run.

```bash
# Report from the most recent run (auto-generated by BRAINSTools_run_tests.sh)
../../Utilities/misc/bt-report-tests.sh                        # default: TestResults/latest.log

# Report from a specific log file
../../Utilities/misc/bt-report-tests.sh TestResults/20260402T084500.log

# Show top-30 slowest instead of top-20
../../Utilities/misc/bt-report-tests.sh -n 30

# Also write a tab-separated raw data file (num, time, status, name)
../../Utilities/misc/bt-report-tests.sh -o /tmp/results.tsv
```

### Why PROCESSORS 3 matters

Each BRAINSTools test CMakeLists sets `PROCESSORS 3` so CTest knows each test
consumes 3 logical CPUs (2 ITK threads + 1 for OS/overhead). CTest then
schedules at most `floor(hw_cpus / 3)` tests concurrently without
oversubscription — no manual thread calculation required.

### Direct ctest (when needed)

```bash
# Release build — normal use
cd build/BRAINSTools-Release-EPRelease-build
ctest -j8 --output-on-failure
ctest -R BRAINSFit --output-on-failure
ctest -N              # list tests without running
ctest -D Experimental # submit to CDash

# Debug build — only for diagnosing a specific failure
cd build/BRAINSTools-Debug-EPRelease-build
ctest -R FailingTestName --output-on-failure
```

## Test Organization

Each tool has a `TestSuite/` or `test/` subdirectory:

```
ToolName/TestSuite/
├── CMakeLists.txt
└── ToolNameTest.cxx            # or ToolNameGTest.cxx for GoogleTest
```

Tests are registered in `TestSuite/CMakeLists.txt` using:
```cmake
# Standard CTest registration
add_test(NAME TestName
  COMMAND ToolName --arg1 val1 --inputVolume ${INPUT_DATA}/image.nii.gz
                   --outputVolume ${TEMP}/output.nii.gz
)

# With ExternalData
ExternalData_Add_Test(BRAINSToolsFetchData
  NAME TestName
  COMMAND ToolName
    DATA{${TestData_DIR}/Input/image.nii.gz}
    ${TEMP}/output.nii.gz
)
```

## ExternalData System

Large test data is not stored in Git. BRAINSTools uses the ITK ExternalData system:

- Data files are referenced via content-hash sidecar files (`.md5` or `.sha512`)
- The `BRAINSToolsFetchData` target downloads data on demand during build
- `DATA{path/to/file}` in CMakeLists resolves to the hash-addressed store

To add new test data:
1. Place the file in `TestData/` or the tool's `TestSuite/` directory
2. Run `Utilities/BRAINSMakeMD5SigFileAndMoveData.py` to generate the hash
3. Upload the data: `Utilities/UploadBinaryData.sh`
4. Commit the hash file (not the data itself)

## Test Output Directory

Tests write output to a temporary directory. The CMake variable is
`${TEMP}` or `${CMAKE_CURRENT_BINARY_DIR}/Temp`. Never hardcode paths.

## CDash Dashboard

Test results are submitted to CDash. To submit from a local build:
```bash
ctest -D Experimental
```
